# AOMS Web — ATM Operations Management System (Next.js)

Вторая версия интерфейса руководителя/администратора — на React/Next.js
вместо Streamlit. Реализовано за сегодняшнюю сессию, с прицелом на то,
чтобы закрыть главную боль — ручной ежедневный ввод банкоматов.

## Что готово и реально протестировано сейчас

- **Dashboard** — реальные счётчики из базы
- **Настройки** — KPI по умолчанию, антифрод, расписание, анализ координат
- **Банкоматы** — список с поиском и фильтрами, карточка, массовое назначение категории/статуса
- **Категории и статусы работы** — расширяемые справочники
- **Сотрудники** — список, добавление, персональная Telegram-регистрационная ссылка
- **Машины** — экипаж из 2 сотрудников
- **Импорт из Excel** — 996 банкоматов, 29 районов, координаты не перепутаны
- **Расписание** — район(ы) + KPI назначаются вручную на каждый экипаж и день,
  можно несколько районов на одну машину в один день
- **Маршруты** — строятся из расписания, банкоматы категории «Внутри здания»
  и неподтверждённых статусов исключаются автоматически, ссылка на Яндекс
  Карты формируется без платного API-ключа (публичный URL-протокол rtext=)
- **Telegram — код готов, но живая отправка не проверена** (см. ниже)
- **Анализ координат** — банкоматы ближе 30 м (порог из настроек) группируются
  через union-find (транзитивно, как в примере ТЗ), решения «объединить/не
  объединять» сохраняются и не теряются при пересчёте. На реальных 996
  банкоматах: 167 групп найдено за ~90 мс
- **Отчёты** — план по расписанию/маршрутам (дата, машина, районы, KPI,
  банкоматов в маршруте, % от KPI), фильтры по дате/машине, **реальный
  экспорт в .xlsx** (проверил — файл открывается, данные корректны)
- **Проблемные банкоматы** — ручная регистрация проблемы (пока без фото —
  загрузка появится вместе с мобильным интерфейсом сотрудника), передача
  технику переиспользует уже готовый Telegram-канал `notifyTechnician`

## Деплой на Railway — пошагово

### Шаг 0 — что уже подготовлено в проекте

- `railway.toml` — команды сборки/запуска, healthcheck
- `.nvmrc` (22.11.0) — Railway (Railpack) поставит нужную версию Node
  (обязательно 22.5+, иначе `node:sqlite` не заработает)
- `npm run start` теперь слушает порт из `$PORT` — Railway сам решает,
  на каком порту работать снаружи, и передаёт это через переменную окружения

### Шаг 1 — выложить код в GitHub (самый простой путь)

```bash
cd aoms-web
git init
git add .
git commit -m "AOMS MVP"
```
Создайте пустой репозиторий на github.com, затем:
```bash
git remote add origin https://github.com/ваш-логин/aoms-web.git
git push -u origin main
```

*(Альтернатива без GitHub: `npm install -g @railway/cli`, затем
`railway login` и `railway up` прямо из папки проекта — тоже работает,
не буду расписывать оба пути параллельно, выбирайте один.)*

### Шаг 2 — создать проект на Railway

1. [railway.app](https://railway.app) → New Project → Deploy from GitHub repo → выбрать `aoms-web`
2. Railway сам определит Node.js/Next.js-проект через свой билдер **Railpack** (актуальный с марта 2026, пришёл на смену Nixpacks) и начнёт первую сборку — она **упадёт**, это ожидаемо, потому что ещё не подключён постоянный диск (следующий шаг)

**Если сборка падает с пустыми логами / "railpack process exited with an error"** — почти наверняка в репозитории завалялся старый `nixpacks.toml` (устаревший формат конфига для другого билдера) — удалите этот файл, если он есть, `railway.toml` для настройки команд достаточно.

**Если сборка падает из-за отсутствия TypeScript/Tailwind** (Railpack не поставил dev-зависимости) — добавьте в Variables сервиса:
```
RAILPACK_INSTALL_CMD=npm ci --include=dev
```

### Шаг 3 — подключить постоянный диск (обязательно, иначе база будет исчезать при каждом деплое)

1. В проекте на Railway → вкладка сервиса → **Volumes** → **New Volume**
2. Mount path: `/data`
3. Сохранить

### Шаг 4 — переменные окружения

Вкладка **Variables** сервиса → добавить:

```
AOMS_SQLITE_DB_PATH=/data/aoms.db
TELEGRAM_BOT_TOKEN=токен_от_BotFather
TELEGRAM_BOT_USERNAME=имя_бота_без_@
TELEGRAM_GROUP_CHAT_ID=
TELEGRAM_TECHNICIAN_CHAT_ID=
TELEGRAM_MANAGER_CHAT_ID=
```

Группу/техника/руководителя можно добавить позже через `/myid` (см. ниже) — не обязательно знать их сразу.

### Шаг 5 — деплой и публичный адрес

1. **Settings** → **Networking** → **Generate Domain** — Railway выдаст бесплатный публичный адрес вида `aoms-web-production.up.railway.app`
2. Redeploy (или он сам пересоберётся после сохранения переменных)
3. Откройте выданный адрес — должен открыться Dashboard

### Шаг 6 — подключить Telegram-бота к этому адресу

Раз у приложения теперь есть постоянный публичный адрес — **бот работает прямо на Railway, ноутбук для этого не нужен**. Настраиваем вебхук (как и планировалось изначально, просто теперь есть куда его указать):

```bash
curl -X POST https://api.telegram.org/bot<ТОКЕН>/setWebhook \
  -d "url=https://aoms-web-production.up.railway.app/api/telegram/webhook"
```

Дальше — как описано выше: у каждого сотрудника на странице «Сотрудники» появится ссылка для подключения; чтобы узнать chat_id техника/руководителя/группы — напишите боту `/myid`.

### Проверка, что всё сохраняется

Зайдите, добавьте банкомат вручную → откройте вкладку **Deployments** на Railway → нажмите **Redeploy** (пересоздаёт контейнер, но не трогает volume) → банкомат должен остаться на месте. Если пропал — значит `AOMS_SQLITE_DB_PATH` не совпадает с mount path volume, проверьте шаги 3-4.

## Общий контекст про хостинг (для справки, если Railway не подойдёт)

### Vercel не подойдёт этому проекту как есть

Vercel Functions работают в serverless-модели: **файловая система
доступна только на чтение**, кроме `/tmp` (временная папка, которая
стирается между вызовами и уж точно между разными запросами разных
пользователей). Наша БД — файл `data/aoms.db` через `node:sqlite` —
физически не может там жить: либо каждый запрос будет видеть пустую
базу, либо будет падать с ошибкой записи.

Это не мелочь и не обходится конфигурацией — это фундаментальное
свойство serverless-платформ (то же самое верно для Cloudflare Workers,
AWS Lambda и подобных). Чтобы задеплоить на Vercel, нужно **сменить
хранилище** с локального SQLite-файла на управляемую БД (например,
Turso — SQLite-совместимая, есть щедрый бесплатный тариф — или Postgres
у Neon/Supabase через Vercel Marketplace). Наш слой `lib/db.ts`
спроектирован так, что эта замена не потребует переписывать
`lib/settings.ts`, `lib/atms.ts` и остальные сервисы — но сам `db.ts`
придётся переписать, а это отдельная итерация, если вам это нужно.

### Другие варианты (для справки, если Railway когда-нибудь не подойдёт)

Бесплатных тарифов с постоянным диском на середину 2026 года практически
не осталось у крупных платформ — Railway и Fly.io убрали постоянный free
tier. Render даёт бесплатный tier без карты, но «засыпает» через 15 минут
простоя, и не гарантированно включает постоянный диск на бесплатном
плане — проверяйте актуальный прайс, если решите переключиться.

Deploy на Vercel возможен, но потребует замены хранилища на управляемую
БД (Turso — SQLite-совместимая, есть бесплатный тариф — или Postgres).
Наш `lib/db.ts` отделён от остального кода именно для такой замены —
но это отдельная работа, если понадобится.

## Оптимизация маршрута — Яндекс.Маршрутизация (VRP)

Добавил интеграцию с отдельным продуктом Яндекса — **Яндекс
Маршрутизация** (VRP API, `courier.yandex.ru/vrs`), который реально
считает оптимальный порядок объезда точек (не путать с обычным Maps
API — это разные вещи, разная регистрация).

**Что нужно для активации:**
1. Зарегистрироваться на [yandex.ru/routing/vrp/](https://yandex.ru/routing/vrp/)
   — по документации доступ выдаётся не мгновенно, нужно оставить заявку
2. Уточнить актуальный бесплатный лимит в личном кабинете (на момент
   написания — до 50 точек, 3 курьера, но условия меняются)
3. Добавить в `.env`: `YANDEX_ROUTING_API_KEY=ваш_ключ`
4. (Опционально) в «Настройках» → категория «Маршруты» указать координаты
   базы/склада (`depot_latitude`/`depot_longitude`) — это точка, откуда
   машина выезжает. Если не указать — используется первый банкомат маршрута

**Как это ведёт себя без ключа**: всё работает ровно как раньше —
маршрут строится в порядке из базы, `optimized: false`, никаких ошибок.

**Честно про эту интеграцию**: я не могу проверить её на реальном
ответе API — ключа нет, а регистрация не мгновенная. Формат запроса
(SVRP — один курьер на маршрут) взят из официальной документации
Яндекса и должен быть верным. Разбор ответа (порядок точек, расстояние)
я проверил на **симулированном** ответе, повторяющем документированную
структуру, — логика разбора и переупорядочивания точек отработала
верно. Но если реальный ответ Яндекса чуть отличается по структуре —
увидим это по понятной ошибке (`optimizationNote` покажет причину, а
не тихо сломается), и поправим парсинг за одну итерацию, когда у вас
будет ключ и мы сможем посмотреть на настоящий ответ вместе.

## Расписание и Маршруты — как проверено

Прогнал полный сценарий: импорт реального файла → пометил часть
банкоматов одного района категорией «Внутри здания» и статусом
«Работает» → создал расписание на этот район → построил маршрут →
убедился, что банкомат «Внутри здания» **не попал** в маршрут, а
подтверждённые банкоматы **попали**, а непроверенные (статус «Новый
банкомат» по умолчанию) — исключены. Ссылка на Яндекс Карты
сформировалась корректно.

**Важная оговорка про маршрут**: ссылка Яндекс Карт строит маршрут
через точки в том порядке, в котором они есть в базе — это **не**
оптимизированный порядок объезда. Настоящая оптимизация (кратчайший
объезд всех точек) требует платного Router API Яндекса с ключом — это
следующая итерация, если понадобится.

**НЕ протестировано и не может быть протестировано в этой среде**:
живая отправка сообщений через `api.telegram.org`. У меня нет ни
реального токена бота, ни доступа к этому домену из моей песочницы
(она разрешает только ограниченный список доменов, Telegram туда не
входит). Код написан по документации Telegram Bot API (обычные HTTP-
запросы, без библиотек), но "будет ли это реально работать" я
гарантировать не могу — только вы можете это проверить, и вот как:

### Настройка Telegram (сделать один раз)

1. Написать [@BotFather](https://t.me/BotFather) в Telegram → `/newbot` → получить токен
2. В `.env` добавить:
   ```
   TELEGRAM_BOT_TOKEN=ваш_токен_от_BotFather
   TELEGRAM_BOT_USERNAME=имя_бота_без_@
   TELEGRAM_GROUP_CHAT_ID=chat_id_общей_группы (опционально)
   TELEGRAM_TECHNICIAN_CHAT_ID=chat_id_техника (опционально)
   ```
3. Развернуть приложение на публичном HTTPS-адресе (для локальной
   разработки — `ngrok http 3000` или аналог)
4. Один раз выполнить (замените токен и домен):
   ```bash
   curl -X POST https://api.telegram.org/bot<TOKEN>/setWebhook \
     -d "url=https://ваш-домен/api/telegram/webhook"
   ```
5. Зайти на страницу «Сотрудники» → у каждого появится ссылка
   «ссылка для подключения» → сотрудник переходит по ней в Telegram,
   жмёт Start → chat_id привязывается автоматически
6. На странице «Маршруты» кнопка «Отправить экипажу» должна заработать

Если после этих шагов сообщения не приходят — значит в моём коде есть
баг, который я не мог поймать без доступа к реальному API. Дайте знать,
разберём по логам.

## Как разобрать банкоматы без категории после импорта

Excel не содержит категорию — угадывать её по адресу я не стал (это
было бы гаданием, а не данными, и могло бы создать неверную классификацию).
Вместо этого на странице «Банкоматы»:
1. Отфильтруйте список (поиск по адресу/району — например, все банкоматы
   в конкретном ТЦ или бизнес-центре обычно идут рядом в списке).
2. Отметьте чекбоксами нужные строки.
3. Сверху появится панель — выберите категорию (и/или статус), нажмите
   «Применить к выбранным».

Это не автоматизирует категоризацию полностью, но сокращает её с 996
кликов до нескольких десятков массовых операций.

## Импорт Excel — как этим пользоваться

1. Страница «Банкоматы» → кнопка «Импорт из Excel»
2. Выбрать файл (тот же формат, что `Список_банкоматов.xlsx`)
3. Загрузить

Что при этом происходит:
- Банкоматы с уже существующим ID (`TerminalID`) — обновляются (адрес, район, координаты)
- Новые — добавляются со статусом «Новый банкомат» (или «Без ID», если
  ID не найден), но **не помечаются как проверенные** — это осознанно:
  импортированные данные нужно подтвердить (см. BR-040/041 из
  архитектуры), прежде чем банкомат попадёт в маршрут
- Строки-разделители районов (в вашем файле вида «Яшнабадскийский» +
  счётчик) распознаются автоматически и становятся значением района
  для всех банкоматов до следующего разделителя
- История очисток/маршрутов не трогается (модуль истории ещё не
  реализован, но импорт уже спроектирован так, чтобы её не удалять)

## Технологии

- **Next.js 14** (App Router) + React + TypeScript
- **Tailwind CSS** — вёрстка на токенах дизайна из макета (цвета, отступы)
- **node:sqlite** — встроенный в Node.js 22+ модуль SQLite. Выбран
  специально вместо Prisma/better-sqlite3: не требует скачивания
  нативных бинарников с внешних серверов при установке — что упрощает
  сборку в закрытых корпоративных сетях банка. Отмечен Node.js как
  экспериментальный API — если это станет проблемой, слой `lib/db.ts`
  можно заменить на `better-sqlite3` без изменения остального кода.

## Архитектура

```
app/               — страницы (Server Components) + Server Actions
lib/db.ts          — низкоуровневый доступ к SQLite (аналог IStorageAdapter)
lib/entity.ts      — общие утилиты (id, даты)
lib/settings.ts    — сервис настроек (бизнес-логика + значения по умолчанию)
lib/categories.ts  — сервис категорий банкоматов
lib/statuses.ts    — сервис статусов работы банкоматов
lib/atms.ts        — сервис банкоматов
```

Чтобы сменить БД на PostgreSQL в будущем — меняется только `lib/db.ts`
(реализовать те же 5 функций: readAll/findById/insertRow/updateRow/
deleteRow через `pg` вместо `node:sqlite`). Ни `lib/settings.ts`, ни
`lib/atms.ts`, ни страницы в `app/` менять не придётся — тот же принцип,
что был в Python-версии с `IStorageAdapter`.

## Установка и запуск

Требуется **Node.js 22.5 или новее** (из-за `node:sqlite`).

```bash
node --version   # должно быть >= 22.5.0
npm install
npm run dev
```

Откроется на `http://localhost:3000`. База данных `data/aoms.db`
создастся автоматически при первом обращении к любой странице —
отдельного шага инициализации не нужно.

## Как проверить, что работает

1. **Dashboard** — цифры сразу реальные (не «—», как было в Streamlit-заглушке).
2. **Настройки** — сохраните значение, обновите страницу (F5) — должно остаться.
3. **Банкоматы** — добавьте банкомат, поставьте категорию «Внутри здания» —
   он получит статус, который в будущем модуле «Маршруты» исключит его
   из автопостроения маршрута (флаг `excludedFromRouting` уже в БД).
4. Полностью остановите (`Ctrl+C`) и снова `npm run dev` — данные должны сохраниться,
   они лежат в `data/aoms.db`, а не в памяти процесса.

Всё это я уже проверил здесь: собрал `npm run build` (проходит
TypeScript-проверку и линт), поднял `npm run start` и прогнал настоящие
HTTP-запросы (создание/поиск/удаление банкомата через POST, не через
прямой вызов функций) — БД реально создаётся, таблицы появляются сами,
данные переживают перезапуск сервера.

## Известные ограничения этой версии (честно, без прикрас)

- `node:sqlite` — экспериментальный API Node.js. Стабилен для MVP, но
  Node может изменить его в будущих версиях без обратной совместимости.
- Next.js 14.2.35 (актуальная патч-версия на момент сборки) имеет ряд
  известных уязвимостей средней/высокой критичности, но все — в
  сценариях, которых этот проект не использует (Image Optimizer с
  `remotePatterns`, `middleware`, `i18n`-роутинг). Перед реальным
  production-развёртыванием — обновить до Next.js 15/16.
- Шрифты — системные (Georgia/Segoe UI/Consolas), не Fraunces/Public
  Sans/JetBrains Mono из макета: сборка не должна зависеть от
  Google Fonts CDN, особенно в закрытой банковской сети. Если внешний
  доступ к fonts.googleapis.com разрешён — шрифты можно подключить
  через `next/font/google` в `app/layout.tsx`.
